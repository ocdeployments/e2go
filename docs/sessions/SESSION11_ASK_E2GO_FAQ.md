## CONTEXT — WHAT THIS IS AND WHY

A public, unauthenticated, AI-powered Q&A widget embedded as a section on
the landing page. Visitor types ANY E-2-related question, gets a direct,
brief, conversational answer. No login required. This is the "try before
you buy" hook for people arriving from Reddit/Facebook E-2 discussions —
if it answers their specific question well, they explore the rest of the
app.

**Model:** `xiaomi/mimo-v2.5` via OpenRouter (OPENROUTER_API_KEY) — this is
the same model Claude Code agents currently use, per owner confirmation.
Follows the existing API key routing rule (OPENROUTER_API_KEY for all app
AI features — generation-engine.ts's ANTHROPIC_API_KEY is untouched).

**Conversion mechanism:** soft CTA after every answer — quiz link, framed
naturally (not pushy).

**Tone:** direct, brief, conversational — NOT the paragraph-length prose
of the source E2_Answers docs. Source docs are the KNOWLEDGE, not the
OUTPUT FORMAT.

## ARCHITECTURE — THREE-LAYER RETRIEVAL

### Layer 1 — Pre-answered corpus (pgvector, primary, near-free)
355 Q&A pairs from `docs/E2_Answers_Part1_Eligibility_Investment.md`
through `Part6_EdgeCases_Alternatives_AppLogic.md`. Embed all 355 QUESTIONS
(not full answers — match on question similarity) into a `pgvector` column
on a new Supabase table. Incoming query → embed → cosine similarity search
→ if top match exceeds threshold (start at 0.80, tune during testing),
retrieve that Q&A's ANSWER as context for Layer 1's response.

### Layer 2 — Broader knowledge base (pgvector, secondary)
If no Layer 1 match clears threshold, fall back to retrieval over the
broader KB docs (E2_Engine_Knowledge_Base_June3_2026.md,
E2Pathway_Vol1-4, Global Consulate Intelligence reports,
module3_denial_audit.md). Chunk and embed these similarly. Lower
confidence — used as context, not verbatim answer.

### Layer 3 — General model knowledge (fallback)
If neither layer produces a confident match, `xiaomi/mimo-v2.5` answers
from its own knowledge, scoped by the legal-boundary system prompt (Step
4 below). Still scoped to E-2/immigration topics — see scope guard.

### Generation step (applies to ALL layers)
Regardless of which layer supplied context, EVERY response is generated by
`xiaomi/mimo-v2.5` via OpenRouter — the retrieved Q&A/KB content becomes
CONTEXT in the prompt, and the model's job is to produce a DIRECT, BRIEF,
CONVERSATIONAL rewrite (2-4 sentences typically) — never dumping the
source's full paragraph verbatim. This keeps tone consistent regardless of
which layer answered.

## STEP 1 — DATABASE: EMBEDDINGS TABLES (pgvector)

Per RULE 4 (never destructive, IF NOT EXISTS, RLS enabled):

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS faq_qa_corpus (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_file TEXT NOT NULL,        -- e.g. 'E2_Answers_Part1...'
  question_number INT,               -- Q1, Q2, etc.
  question TEXT NOT NULL,
  answer TEXT NOT NULL,               -- full source answer (context for generation)
  sources TEXT,                       -- citation block from source doc
  question_embedding vector(1536),    -- dimension depends on embedding model chosen
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE IF NOT EXISTS faq_kb_chunks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_file TEXT NOT NULL,
  chunk_text TEXT NOT NULL,
  chunk_embedding vector(1536),
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Optional: log queries for cache-hit analysis and future seed expansion
CREATE TABLE IF NOT EXISTS faq_query_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  query_text TEXT NOT NULL,
  matched_layer TEXT,  -- 'corpus' | 'kb' | 'fallback'
  matched_question_id UUID REFERENCES faq_qa_corpus(id),
  similarity_score FLOAT,
  response_text TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX IF NOT EXISTS faq_qa_corpus_embedding_idx
  ON faq_qa_corpus USING hnsw (question_embedding vector_cosine_ops);
CREATE INDEX IF NOT EXISTS faq_kb_chunks_embedding_idx
  ON faq_kb_chunks USING hnsw (chunk_embedding vector_cosine_ops);

ALTER TABLE faq_qa_corpus ENABLE ROW LEVEL SECURITY;
ALTER TABLE faq_kb_chunks ENABLE ROW LEVEL SECURITY;
ALTER TABLE faq_query_log ENABLE ROW LEVEL SECURITY;
-- These are read by server-side API routes only (service role) —
-- no public RLS policies needed; deny-all is correct default.
```

## STEP 3 — API ROUTE: `/api/faq/ask`

`src/app/api/faq/ask/route.ts` — public (no auth required), POST.

### Flow
1. **Rate limit check first** (Step 5) — reject before any expensive work
2. **Scope guard** — quick check the query is plausibly E-2/immigration-
   related. If wildly off-topic (e.g. "write me a poem about cats"),
   respond with a brief, friendly redirect WITHOUT calling the LLM:
   "I'm focused on E-2 visa questions — happy to help with anything in
   that space!" (Use simple heuristics first — keyword presence, length —
   before considering an LLM call for this check; the check itself should
   be near-free.)
3. **Embed the query** (same embedding provider as Step 2)
4. **Layer 1 search** — cosine similarity against `faq_qa_corpus`. If top
   match ≥ threshold (start 0.80) → that Q&A's answer becomes PRIMARY
   context
5. **Layer 2 search** (if Layer 1 below threshold) — similarity against
   `faq_kb_chunks`, top 2-3 chunks as context
6. **Generation call** — `xiaomi/mimo-v2.5` via OpenRouter, with:
   - System prompt (Step 4)
   - Retrieved context (Layer 1 answer OR Layer 2 chunks OR none)
   - The user's question
   - Instruction: respond in 2-4 sentences, conversational, direct
7. **Log** to `faq_query_log` (async, don't block response)
8. Return: `{ answer: string, showCTA: boolean }` — `showCTA` always
   `true` per the soft-CTA-after-every-answer decision, but kept as a
   field for future tuning

### Streaming
Per "answers cannot be laggy" — use streaming response (SSE or
ReadableStream) so the answer appears progressively, not after a full
round-trip wait. `xiaomi/mimo-v2.5` via OpenRouter supports streaming —
confirm and wire it.

## STEP 4 — SYSTEM PROMPT (legal boundary + tone)

Adapt from Spec3's Universal System Prompt (generation-engine.ts) for THIS
context — same legal boundary, different tone target:

```
You are E2go's E-2 visa information assistant. You help people understand
the U.S. E-2 Treaty Investor Visa process.

TONE: Direct, brief, conversational — like a knowledgeable friend, not a
legal document. 2-4 sentences typically. No headers, no bullet lists
unless the question genuinely requires a short list. Get to the point.

CONTENT RULES:
- Use the provided context (if any) as your primary source
- Never say "you are eligible", "you qualify", "this meets the
  requirement" — these are legal conclusions. Say what the rules/process
  generally are, not whether THIS PERSON meets them.
- If you don't have confident information on something, say so plainly —
  "I don't have a confident answer on that specific point" — and suggest
  they explore further or consult an attorney for their specific case.
- Never fabricate specific dollar amounts, dates, or processing times not
  in your context.
- End with a brief, natural mention that e2go's eligibility quiz can give
  them a personalized picture — vary the phrasing, don't be repetitive or
  pushy.

SCOPE: E-2 visa and closely related topics (treaty countries, investment
requirements, business types, family/dependents, application process,
green card pathways from E-2, cross-border tax basics, renewals, denials).
If asked something unrelated, politely redirect.

DISCLAIMER: This is informational only, not legal advice. e2go is not a
law firm.
```

## STEP 5 — RATE LIMITING (Upstash Redis)

1. Set up Upstash Redis (free tier) — add `UPSTASH_REDIS_REST_URL` and
   `UPSTASH_REDIS_REST_TOKEN` to `.env.local` and Vercel env vars
2. Install `@upstash/redis` and `@upstash/ratelimit`
3. Rate limit `/api/faq/ask`: suggest **10 requests per IP per 10 minutes**
   (generous enough for genuine exploration, tight enough to bound cost) —
   confirm/tune with owner if needed
4. On rate limit exceeded: friendly message, not a raw 429 — "You've asked
   a lot of questions! Take a look at our eligibility quiz for a fuller
   picture, or try again in a few minutes." (turns the limit into a CTA)
5. Document the new env vars in `.env.example`

## STEP 6 — FRONTEND: LANDING PAGE WIDGET

New section on the landing page (`HomeClient.tsx` or wherever sections are
composed) — per owner: embedded as a widget/section, not a separate route.

### Design requirements
- Obsidian Gold design system — Cormorant Garamond headings, DM Sans body,
  `#0a0a0a`/`#C9A84C`/`#f5f0e8` tokens, zero border-radius, no
  glassmorphism, grain texture if consistent with hero treatment
- **Highly interactive, highly welcoming** — this is the "wow" moment for
  cold traffic. Use Lazyweb research: "AI chat widget dark luxury landing
  page", "conversational search hero section". Use Magic MCP to generate
  the component.
- Streaming response renders progressively (typing/streaming indicator —
  subtle, on-brand, NOT a generic spinner)
- Suggested starter questions (3-4 chips/buttons) pulled from common
  E2_Answers topics — e.g. "Can my spouse work in the US?", "What
  countries qualify?", "How much do I need to invest?" — clicking one
  submits it immediately, lowering the barrier for first-time visitors
- After each answer: the soft CTA — natural sentence + button to `/quiz`
  (e.g. "Curious how this applies to you? → Take the 2-minute eligibility
  quiz")
- Mobile-first — this is where Reddit/Facebook traffic lands, often on
  phones
- Empty/loading/error states all styled per design system — no default
  browser styling ever visible

### Multi-turn?
Single-question-at-a-time is sufficient for v1 — no conversation history
required. Each question is independent. (Note as a possible future
enhancement, don't build it now.)

## STEP 7 — SCALABILITY NOTES (apply broadly, verify for this feature)

- Vercel serverless functions scale horizontally automatically — the
  concern is NOT "too many requests crash the server", it's (a) Supabase
  connection exhaustion, (b) OpenRouter/embedding API costs, (c)
  cross-instance rate limiting (Step 5 solves this)
- Use Supabase connection pooling (confirm `pgbouncer`/transaction mode is
  configured — check existing Supabase client setup, this affects ALL
  routes not just this one, note if it's missing as a broader finding)
- pgvector HNSW index (Step 1) keeps similarity search fast even as
  `faq_qa_corpus`/`faq_kb_chunks` could grow later
- Log `faq_query_log` asynchronously (fire-and-forget, don't await before
  responding) so logging never adds latency to the user-facing response

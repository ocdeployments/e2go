## FIX A — Bracket-Highlighting Regex

### Problem

The .docx builder's bracket-highlight logic (find it in
docx-builder.ts) appears to expect a literal `[BRACKET FORMAT]`
string per the original spec language. Real generated documents use
DESCRIPTIVE brackets instead. Examples actually present in
cover_letter's content_text for this application:

```
[passport number from Tab A]
[formation date from Tab E]
[EIN from Tab E]
[net worth figure from Tab H]
[percentage]
[enrollment projection from Tab K]
[timeframe]
[number]
[property / family / financial ties from Tab A]
```

ds160_reference has many more, including bare `[Enter from Tab A]` /
`[Enter planned arrival date]` style placeholders, and also uses
`TO CONFIRM` (not bracketed) in some fields — TO CONFIRM is OUT OF
SCOPE for this fix, only bracketed `[...]` text.

### Fix

1. Find the current highlight regex/string-match in docx-builder.ts.
2. Replace with a pattern matching ANY bracketed text: `\[[^\[\]]+\]`
   (matches `[...]` where the contents don't themselves contain
   brackets — avoids greedy multi-bracket matches).
3. BEFORE finalizing — grep all 6 documents' content_text for this
   application for any LEGITIMATE use of square brackets that is NOT
   a placeholder (e.g. citations like "[1]", or markdown-style
   references). If any exist, the regex may need refinement (e.g.
   require the bracket contents to be >2 words, or contain "Tab",
   "from", "Enter", "Confirm" — but prefer the simple `\[[^\[\]]+\]`
   first and only add exclusions if grep shows a real false-positive
   case).
4. Apply highlight (yellow background, per spec) to each match when
   generating the .docx.

## FIX C — Markdown Table → Word Table (investment_proof)

### Problem

investment_proof's content_text (after Piece 1's recovery) contains a
markdown table:

```
| Category | Amount | Purpose |
|----------|--------|---------|
| Franchise Fee | $1,000 | Initial franchise fee payable to Kumon North America |
| Leasehold Improvements | $48,000 | Build-out of the Cedar Park center space |
| Equipment | $22,000 | Furniture, computers, and center fixtures |
| Curriculum Materials | $18,000 | Kumon worksheets and instructional materials |
| Professional Fees | $15,000 | Legal, accounting, and formation costs |
| Marketing | $10,000 | Pre-opening enrollment and brand marketing |
| Working Capital | $71,000 | Pre-opening expenses and initial operations |
```

This currently renders as literal pipe-and-dash text in the .docx
output, not as a Word table.

### Fix

1. Find where content_text is split into paragraphs/runs for the
   .docx (docx-builder.ts).
2. Add detection: a contiguous block of lines where EVERY line starts
   and ends with `|`, AND the second line matches a separator pattern
   (`|---|---|...` or `|----|----|...` — dashes between pipes) is a
   markdown table.
3. When detected: parse into rows/cells (split each line on `|`, trim
   whitespace, drop empty first/last elements from the split). First
   row = header. Skip the separator row. Remaining rows = data.
4. Generate a Word Table (docx library's Table/TableRow/TableCell —
   check what's already imported/used elsewhere in docx-builder.ts
   for consistency) with header row styled distinctly (bold, or
   per existing heading conventions in the builder) and data rows
   below.
5. Continue normal paragraph processing for content before/after the
   table block.

### Scope check before starting

grep all 6 documents' content_text for this application for `|---`
or similar separator patterns — confirm investment_proof is the ONLY
document with a markdown table (expected), so you know the full
scope of what this fix needs to handle. If other documents ALSO have
tables, note this but the same fix should handle them generically.

## CONTEXT — THE STRATEGY: THREE LAYERS, NOT TWO

Session 5 found that the LLM, left entirely unguided, STILL produced a
strong bridge for Chen (RBC → Kumon) because the connection was
obvious. But "at risk when the transfer isn't obvious" (Test B —
warehouse supervisor → cleaning franchise) is exactly the case the
original requirement was written for.

There is a THIRD layer, upstream of the two below: if Layer 1's scoring
(Step 2) finds experience_score = WEAK with few/no confirmed dimensions,
the system should ALSO trigger a TARGETED FOLLOW-UP QUESTION — per
Spec2 Category B's `experience_score < ADEQUATE and
follow_up_not_completed` — designed specifically to surface MORE raw
material before Layer 1's framing call (Step 3) and Layer 2's backstop
(Step 5) have to work with whatever's already on file. This is Step 0
below, and it runs BEFORE Step 3 in the actual pipeline order (gap →
targeted question → richer data → framing call).

Session 5 Piece 2 found this is currently PARTIAL: business-type
extraction in the follow-up question generator looks for keys that
don't exist for Chen/most seeded apps, falls back to "the selected
business type" generically, and there are no business-category-specific
question templates beyond the three hardcoded examples (care, food,
education) in Spec2. Step 0 fixes this using the SAME
business-operational-needs reference table (Step 1) that the rest of
this session builds — one reference table, three consumers.

This session builds THREE layers so no single one has to be perfect alone:

**LAYER 0 — Targeted follow-up question (the "more data" trigger):**
When Step 2's scoring shows experience_score = WEAK with few confirmed
dimensions, generate a follow-up question SPECIFIC to this applicant's
gap and this business category's operational needs — not the generic
"the selected business type" fallback Session 5 found. Responses feed
back into `loadApplicationAnswers()` for Layer 1.

**LAYER 1 — Real structured pipeline (the "rails"):**
Actually compute the 9 experience dimensions from real data, and make a
REAL, FOCUSED AI call (`generateFramingDecisions()`) whose ONLY job is:
"given this applicant's actual background facts and this business
type's operational needs, identify the strongest transferable
connections and write them as explicit framing instructions." This is a
small, cheap, structured call — not the big document-generation call.
Its output becomes `framing_decisions.experience` in the case brief,
which is interpolated into the Document 1 prompt per the original spec.

**LAYER 2 — Hardened generation prompt (the "backstop"):**
Independently of Layers 0/1, strengthen the Document 1 Section II prompt
itself so that even if `framing_decisions.experience` came back thin or
empty (Layer 1 partial failure, AI call error, fixture gap, etc.), the
BIG generation call ALSO carries an explicit instruction to perform this
transferable-skills analysis itself, using the same reasoning framework.

The three layers are NOT identical and don't run redundantly — Layer 0
runs first (before generation, during the follow-up conversation) and
exists to get MORE raw material on file. Layer 1 runs next, is cheap,
focused, and its output FEEDS Layer 2's prompt. Layer 2 ALSO has standing
instructions to do this work even without Layer 1's input. If Layer 1
produces a strong framing_decisions entry, Layer 2 uses it directly
(higher quality, more specific). If Layer 1 comes back thin/empty for any
reason, Layer 2's standing instruction still fires.

This is the layered reliability model. No single layer alone is claimed
to be perfect — together, the failure modes don't overlap.

## STEP 3 — generateFramingDecisions() — REAL AI CALL (Layer 1, Part B)

This is the core of Layer 1. Replace the hardcoded `[]` return with a
real, focused AI call.

### When it runs
After Step 2's scoring. Per Spec1 Category C: when
`overall_experience_score = WEAK` (or ADEQUATE with notable gaps —
use judgment, but WEAK is the must-run case).

### The call itself

Small, cheap, structured — NOT the big document generation call. Use
OpenRouter per the API key routing rule (this is an "app AI feature," not
document generation — ANTHROPIC_API_KEY stays exclusive to
generation-engine.ts per CLAUDE_CONTEXT).

INPUT to the call:
- Applicant's ACTUAL background facts: employment history, education,
  follow-up responses (`mentioned_experiences`, `content_signals`),
  writing sample excerpts if relevant
- Target business category + Step 1's operational needs list for that
  category
- The 9 dimension scores from Step 2 (so the AI knows what's ALREADY
  confirmed vs. what gaps need creative framing)

PROMPT STRUCTURE (build this prompt — this is the actual deliverable of
this step):
```
You are analyzing an E-2 visa applicant's background to identify
genuine, honest connections between their experience and the
operational demands of the business they plan to run.

APPLICANT BACKGROUND:
[employment history, education, follow-up responses — verbatim facts]

TARGET BUSINESS: [category]
OPERATIONAL DEMANDS OF THIS BUSINESS:
[Step 1's list for this category]

EXPERIENCE GAPS IDENTIFIED:
[dimensions scored WEAK/ABSENT from Step 2]

TASK:
For each operational demand, identify whether the applicant's
background contains a genuine, supportable connection — even if
indirect (e.g. household management, volunteer coordination, a
different industry's analogous function, military service, raising
children, community leadership).

For each connection found, output:
{
  "operational_demand": "...",
  "applicant_evidence": "the specific fact from their background",
  "connection": "the specific skill/capability this demonstrates",
  "framing_instruction": "a 1-2 sentence instruction for how to write
    this connection into the document — following the BRIDGE pattern:
    'My experience [X] directly prepares me to [Y] because [specific
    connection]'"
}

Only include GENUINE connections. If no honest connection exists for a
given demand, omit it — do not fabricate. Output valid JSON array only,
no other text.
```

OUTPUT: array of these framing objects → this becomes
`framing_decisions.experience` in the case brief (replacing the
hardcoded `[]`).

### Error handling — this matters for the dual-layer model
If the AI call fails (timeout, malformed JSON, empty array genuinely
returned because no connections exist) — `framing_decisions.experience`
is `[]` or absent. This is FINE and EXPECTED sometimes (Layer 2 is the
backstop). Do NOT retry indefinitely or block generation on this call's
success. Log failures for monitoring but proceed.

failure/empty → Layer 2's standing instruction (interpolated with Step
